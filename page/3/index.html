平台无关是如何实现的

java源码首先被编译成字节码，java提供了各种不同平台上的虚拟机,再由不同平台的JVM进行解析，Java语言在不同平台上运行时不需要进行重新编译，Java虚拟机在执行字节码的时候，把字节码转换成具体平台上的机器指令。

为什么JVM不直接将源码解析成机器码去执行呢？

如果我们直接将源码解析成机器码去执行，每次执行都需要进行语法的检查，这样整体的性能受到影响，引入中间字节码后，能够保证在被编译成字节码之后，多次执行程序是不需要执行各种校验的。2. 同时可以脱离java的束缚，有Gove，Salc来生成字节码，同样可以被jvm进行调用执行，进而也能增加平台的兼容扩展能力

JVM如何加载.class文件

Java虚拟机：是一种抽象化的计算机，JVM有自己完善的硬件架构，如处理器，堆栈，寄存器等，还具有相应的指令系统。JVM屏蔽了与具体平台相关的信息，使得Java程序只需在java虚拟机上运行字节码就可以在多个平台运行代码。屏蔽不通过操作系统的不同并且减少基于原始语言的复杂性

ClassLoader的作用：加载class文件到内存中 ，具体能不能执行得看Execution Engine

Execution Engine 解释器：负责对命令进行解析，解析完成后，就交给操作系统去执行

Native Interface ：融合不同开发语言的原生库为java所用，具体是 Native Method Stack 登记native 方法，在执行时加载Native Libraies

Runtime Data Area : 就是内存数据模型，我们所写的程序都会被加载到这里，之后才会被运行

JVM主要由Class Loader Runtime Data Area ，Execution Engine ， Native Interface 这四个部分组成，主要通过class loader ，将符合其格式要求的class文件加载到内存里，并通过Execution Engine 解析class文件里面的字节码，提交给操作系统去执行

谈谈ClassLoader

  • 概述

    ClassLoader 主要工作在Class加载阶段，将Class文件里的二进制数据流转载进系统，然后交给Java虚拟机进行连接，初始化等操作

  • ClassLoader的种类

    1. BootStrapClassLoader :C++编写，加载核心库java.*
    2. ExtClassLoader: java编写，加载扩展库javax.*
    3. AppClassLoader: java编写，加载程序所在目录，环境变量classpath

  • 自定义ClassLoader: Java编写，定制化加载,可能不在系统的classpath的范围内，可以不是class文件或jar文件。有很多用途

    findClass 就是根据名称或位置来加载字节码，会调用defineClass来解析定义字节流，返回Class对象。

    通过自定义的ClassLoader，我们不仅可以通过自定义路径来加载class文件，还可以从网上获取二进制的流，来生成二进制的类，还可以对某些敏感的class文件进行加密，在findclass里面对其进行解密，也可以修改二进制的流，给类添加信息(ASM字节码生成技术。也可以做为AOP的联想)

  • ClassLoader的双亲委派机制

    从前面了解到，不同的classLoader加载类的方式和路径有所不同，为了分工，各自负责各自的区块，各司其职，所以存在一个机制，让他们相互写作，形成一个整体，这就是双亲委派机制

    S首先自底而上的检查类是否已经加载，从自定义的类加载器开始找有没有加载过该类，如果加载过直接返回，如果没有，就委派给他的parent也就是App ClassLoader，如果也没有，就找Extension ClassLoader，如果没有，就到最顶层Bootstrap ClassLoader。如果也没有，就去rt.jar或者-Xbootclasspath指定的jar包。如果还没有，就委托给Extension ClassLoader 去ext目录下的jar文件中找，或者去-Djava.ext.dirs指定目录下的jar包.

  • 双亲委派机制可以防止多份同样字节码的加载，如果不适用，每个类都各自加载字节码文件，内存中就会有重复的字节码文件存在,使用双亲委派机制，只会加载一次该类

类的加载方式

  • 隐式加载 new 可以使用构造方法给类初始化值

loadClass和forName的区别

  • 都能获得任意一个类的所有方法和熟悉
  • loadClass的没有链接过程（resolve=false），而forName有链接过程（initial = true）
  • Class.forName得到的class是已经初始化完成的 会执行static
  • Class.loadClass得到的class是还没有链接的 不会执行 static
  • forName的使用场景: jdbc.Driver 中有static代码
  • loadClass的是使用场景: SpringIoC 使用的懒加载技术，由于loadClass不用初始化，所以使用loadClass来加快懒加载速度，把类的初始化工作留到类的使用的时候在去做

    • 解析 JVM将常量池内的符号引用转换为直接引用
  • 初始化
    • 执行类变量赋值和静态代码块

JVM的内存模型

  • 程序计数器:

    • 是一块较小的内存空间,当前线程所执行字节码的行号指示器，在虚拟机的概念模型里，字节码解释器工作时，就是

  • 改变计数器的值来选取下一条需要执行的字节码指令，包括分支，循环，跳转，异常处理,线程恢复等基础功能，都需要依赖这个计数器来完成

    • 在任何一个确定的时刻，一个处理器只会执行一条线程中的指令，因此，为了线程切换后能回复到正确的执行位置，每条线程都需要独立的程序计数器，各条线程之间的计数器互不影响，独立存储，我们称类内存区域为线程私有的内存

  • 对java方法计数，如果是Native方法则计数器值为Undefined

    • 由于只是记录行号，不会发生内存泄漏的问题

  • 程序计数器是逻辑计数器，而非物理计数器，为了线程切换后都能回到原来的位置，每个线程都有一个独立的程序计数器，是线程独立的，并且只为Java方法计数，native方法对应的计数器是undefin，使用程序计数器不用担心发生内存泄漏的问题

  • Java虚拟机栈

    • Java方法指向的内存模型

    • 每个方法被执行时都会创建一个栈帧，即方法运行时的基础数据结构，用来存储局部变量表，操作栈，动态连接，返回地址

    • 每个方法执行中，对应虚拟机栈帧，从入栈到出栈的过程，java虚拟机栈用来存储栈帧，而栈帧持有局部变量，和部分结果，与参与方法的部分返回，当方法调用结束时，帧才会被销毁。

    • 虚拟机栈包含了单个线程每个方法执行的栈帧，栈帧则存储了局部变量表，操作数栈，动态连接，和方法出口，等信息

      • 局部变量表：包含了方法执行过程中的所有变量，包括this引用，所有方法参数，其他局部变量，包括布尔值，byte，char，long short int double 等。

      • 操作数栈：在执行字节码指令过程中会用到，这种方式类似于原生cpu寄存器，大部分jvm字节码把时间花费在操作数栈的操作上，包括入栈，出栈，复制，交换，产生消费变量的操作，因此，局部变量数组和操作数栈之间的交换变量指令操作通过字节码频繁执行

        我们都知道栈是一个后进先出的数据结构，因此当前执行的方法在栈的顶部，每次方法调用时，一个新的栈帧创建，并压栈到栈顶，当方法正常返回或抛出未捕获的异常时，栈帧就会出栈，除了栈帧的压栈和出栈，栈不能被直接操作。局部变量表给操作数栈做一个数据支撑

  • Java递归为什么会造成stackoverflow 异常

    • 当线程执行一个方法时，就会随之创建一个栈帧，并将加入的栈帧加入到虚拟机栈中，当方法执行完毕的时候，便会对栈帧出栈，因此可知，线程当前执行的方法，所对应的栈帧，必定位于java栈的顶部，而我们函数不断调用自身，每次调用方法都会创建栈帧，保存栈帧状态，将他放在虚拟机栈中，栈帧切换时会切换到最新的方法栈帧中，而我们每个线程的虚拟机栈的深度固定的，递归实现会导致栈深度的增加，每次递归都会往栈里压一个栈帧，如果这一消耗超过了栈允许的最大深度，就会报stackOverFlowError异常，解决这个问题的思路就是限制递归的深度，或者将递归变成迭代

    虚拟机栈也是java虚拟机自动管理的，栈类似于一个集合，但是他是有固定的容量的，是由多个栈帧合起来的，再我们编写代码的时候，当我们允行程序的时候，每调用一个方法，java虚拟机就会分配对应的一块空间，这个空间就是栈帧啦，而当方法调用结束后，对应的栈帧就会自动释放掉，这也是栈的空间不需要gc，而会自动回收的原因啦

    元空间（metaSpace）与永久代（PermGen）的区别

再jdk8之后，将类的元数据放到本地堆内存中，这一块就叫metaSpace，该区域在jdk7及以前是属于永久代的，元空间和永久代都是用来存储class对象的相关信息，包括method 和field等，二者都是方法区的实现，只是实现有所不同，所以说方法区只是一种jvm的规范，在java7之后，原先位于方法区的字符串常量池已被移动到了java堆中，并且在jdk8，元空间替代了永久代。

  • 元空间使用本地内存，而永久代使用jvm内存
  • 理论上本地内存还有多少，meteSpace就能有多大，解决了空间不足的问题，jvm默认会根据运行时的需要动态设置其大小.

Redis 简介

虽然在数据库层也做了缓存，但这种一般针对的是查询的内容，力度比较小，一般只有表中数据没有变更的时候，数据库对应的cash才会发生作用，但这并不会减少业户系统对数据库产生的增删改查的IO压力，因此缓存技术应运而生，该技术实现了对热点数据的缓存，提高响应的速率 ，极大缓解后端数据库的压力。如图是主流应用架构：

为了提高性能，一般在客户端和存储层之间添加一个缓存层，当客户端向后端发送请求的时候，先去缓存层查， 有数据就直接返回。如果缓存层没有相应的数据，就会去存储层查，这就是 穿透 ，如果客户端查到数据了，就回写到缓存层，以便下一次可以直接从缓存层返回，回写的过程就是 回种，回种完成之后，就把数据返回给客户端。 这个架构还能实现熔断机制，当我们发现存储层挂掉，或者没办法提供服务的时候，可以让客户端的请求直接打到缓存上，不管有没有获取到数据，都直接返回，这样就能够在有损的情况下提供服务。

缓存中间件 - Memcache 和Redis 的区别

Memcache：简单易用，代码层次类似于Hash

  • 支持简单的数据类型
  • 不支持数据持久化存储，一旦宕机后，数据是没法保存的
  • 不支持主从同步
  • 不支持分片，指的是将整个数据打碎的过程，将大数据打碎放到多个物理节点上的方案

Redis

  • 数据类型丰富
  • 支持数据磁盘持久化存储
  • 支持主从
  • 支持分片

为什么Reids能这么快

  • 完全基于内存，绝大部分请求是纯粹的内存操作，执行效率高。 是采用单进程，单线程模型的k-v数据库，由C语言编写，将数据储存在内存中，读写数据的时候都不会收到硬盘IO速度的限制，所以速度极快

  • 数据结构简单，对数据操作也简单，Redis不使用表，他的数据库，不会预定义或者强制要求用户对存储的不同数据进行关联，因此性能相比关系型数据库高很多，存储结构就是键值对，类似于hashmap，时间复杂度为O（1）的

  • 采用了单线程，单线程也能处理高并发请求，想多核也可以启动多实例

    ​ 在面对多并发的请求的时候，首先想到的是用多个线程来处理，将IO线程和业务线程分开，业务线程来使用线程池来避免频繁销毁线程池，即便是一次请求，阻塞了也不会影响其他请求。

    ​ 为什么Redis不这样呢？准确的来说，Redis的单线程结构，是指其主线程是单线程的，这里主线程是指IO线程的处理，以及Io对应的相关请求的处理，其外，主线程还负责过期键的处理，复制协调，集群协调等等这些处理IO事件之外的逻辑，会被封装成周期性的任务，由主线程周期性的处理，正因为采用单线程的设计，对于客户端的所有读写请求，都由一个主线程串行的处理，因此，多个客户端对同一个键进行写操作的时候，就不会有并发的问题，避免了上下文切换和锁竞争，使得效率更高

    ​ 单线程可以处理高并发的请求嘛？并发并不是并行，并行是指服务器同一时间可以执行几个事，具有多个计算单元。而并发性IO流，意味着能够让一个计算单元来处理来自多个客户端的流请求，Redis使用单线程配合上IO多路复用，将能大幅度的提升性能

    ​ 制约Redis性能的瓶颈并不是CPU，只是网络等问题导致处理量不能进一步上升，仍然可以在多核服务器上启动多个Redis 实例来利用多核的特性

    ​ 单线程只是在处理网络请求的时候只有一个线程来处理，一个正式的Redis Server 在运行的时候肯定不止一个线程的，例如Redis 在进行持久化的时候，会根据实际情况，以子进程和子线程的方式执行

  • 使用了多路IO复用模型，非阻塞的IO

    Redis 是跑在单线程中的，所有的操作都是按照顺序线性执行的，但是由于读写操作，等待用户输入或者输出，都是阻塞的，所以IO操作在一般情况下都不能直接返回，这会导致某一文件的IO阻塞，进而导致整个进程无法对其他客户端提供服务，IO多路复用就是为了解决这个问题而出现的

多路IO复用原则

FD： File Description ，文件描述符

  • 一个打开的文件通过唯一的描述符进行引用，该描述符是打开文件的元数据到文件本身的映射

传统的阻塞IO模型：

当使用read 或者write 对某个文件描述符fd进行读写时，如果当前fd不可读，或者不可写，整个redis 服务就不会对其他的操作做出响应，导致整个服务不可用，这就是传统常见的阻塞模型，但是他会影响其他fd对应的服务，所以在处理多个客户端应用的时候，往往都不会使用阻塞模型

在IO多路复用模型中，最重要的函数调用就是Select 系统调用

selcct 能够同时监控多个文件的可读可写情况，当其中的某些文件可读或者可写时，Select方法就会返回可读或者可写的描述符个数。用来监听

多路IO复用模型：

Redis采用的IO多路复用函数：epoll/kqueue/evport/select ，会根据编译平台的不同来选择不同的函数，优先选择时间复杂度O1的作为底层实现，以时间复杂度为O（n）的select 作为保底。基于react设计模式监听IO事件，实现文件事件处理器，使用多路IO复用模块，同时监听多个fd，当accept，read，wirte，close文件事件产生时，文件事件处理器就会回调fd绑定的事件处理器，虽然文件事件处理器时在单线程下运行的，但是通过IO多路复用模块的引用,实现了同时对多个fd读写的监控，提高了网络通信模型的性能

I/O多路复用的设计遵循以下原则：

  1. I/O多路复用：当任何文件描述符准备好I/O时告诉我
  2. 在一个或更多文件描述符就绪前始终处于睡眠状态。
  3. 唤醒：哪个准备好了？
  4. 在不阻塞的情况下处理所有I/O就绪的文件描述符。
  5. 返回第一步，重新开始

Redis的数据类型

  • String :最基本的数据类型，最大能存储512M，二进制安全的，可以包含jpg图片，或者序列化的对象，String类型之所以支持存储如此多的数据类型，离不开底层的sds动态字符串

  • Hash:String 元素组成的字典，适用于存储对象

  • List :列表，按照String元素插入顺序排序，你可以添加一个元素到列表的头部，即左边或者尾部 ,可以实现栈的功能，可以做最新消息排行榜的功能

  • Set ：String 元素组成的无序集合，通过哈希表实现，不允许重复

  • Sorted Set : 通过分数来为集合中的成员进行从小到大的排序，分数越小，越往前排

  • 用于技术的Hyp二log log，用于支持存储地理位置信息的Geo

从海量Key里查询出某一固定前缀的Key

  • KEYS pattern : 查找所有符合给定模式pattern的key

    1. 一次性返回所有的key，
    2. key的数量过大，会导致服务卡顿。

  • SCAN cursor [MATCH pattern] [COUNT count]

    1. z 每次执行都只会返回少量数据，不会阻塞服务器

    2. 基于游标的迭代器，命令每次调用都需要获得上一次调用返回的游标作为该次调用的游标参数，以此来延续之前的迭代过程

    3. 单scan指令的游标参数 cursor 被置为0时，服务器会开始一次新的迭代，而服务器向用户返回0的游标时，表示迭代已经结束。

    4. 不保证每次执行都返回某个给定数量的元素，支持模糊查询

    5. 一次返回的数量不可控，只能时大概率符合count参数

    6. 可能后一次的游标比前一次还小，有可能获取重复key，需要去重

如何通过Redis 实现分布式锁

分布式锁是控制分布式系统或不同系统之间共同访问共享资源的一种锁的实现，如果不同系统或者同一个系统不同主机之间共享了某一资源的时候，往往需要互斥来避免彼此干扰，进而保证一致性

分布式锁需要解决的问题

  • 互斥性：任意时刻，只能由一个客户端获取锁，不能同时有两个客户端获取到锁
  • 安全性：锁只能由持有该锁的客户端删除，不能由其他客户端删除
  • 死锁：获取锁的客户端由于宕机而无法释放锁，其他客户端无法获得该锁导致死锁
  • 容错：当节点宕机的时候，客户端仍然能够获取锁和释放锁

如何通过Redis实现分布式锁

  • SETNX key value ：如果key不存在，则创建并赋值，如果key存在，就会失败,在执行某段代码时，先尝试setnx对某个key设值，如果成功，则证明没有其他线程在执行该段代码或者占用该独占资源，这时候我们的线程就可以执行该代码逻辑，如果设值失败，则需要等待直到setnx成功

    1. 时间复杂度：O1
    2. 返回值：设置成功，返回1，设置失败 ，如果存在该值就会失败，返回0

  • 解决SETNX长期有效的问题：EXPIRE key seconds

    1. 设置key的生存时间，当key过期时(生存时间为0)，会被自动删除

    2. 先setnx看看有没有锁，获得status ==1 ，然后执行资源，如果有别的资源想要执行该段代码，发现status = 0，就阻塞直到status = 1

    3. 上面的方法有风险，就是setnx后直接挂掉，没有来得及释放锁，原子性得不到满足

  • 使用SET key value [EX seconds] [PX milliseconds] [NX|XX]

    1. EX seconds :设置键的过期时间为second秒
    2. PX millisecond: 设置键的过期时间为millisecond毫秒
    3. NX：只在键不存在时，才对键进行设置操作
    4. XX：只在键已经存在时，才对键进行设置操作
    5. SET 操作成功返回ok，否则返回nil

Redis如何做RDB持久化

  • RDB快照持久化：保存某个时间点全量数据快照

    1. redis持久化策略

      900s之内如果有一条是写入指令，就产生一次快照

      300s之内如果有10条写入指令，就产生一次快照，如果变动数大于0小于10的话，就会等900s后执行快照

    2. 当设置成yes的时候，表示当备份进程出错的时候，主进程就停止写入的操作，这样做是保护持久化数据一致性的问题，如果系统有完善的监控系统呢，可以关闭，否则开启

    3. 在进行备份的时候将RDB文件进行压缩之后才去做保存，建议设置成no，因为redis本身就是cpu密集型服务器，再开启压缩会带来更多的cpu消耗

  • RDB文件的创建与载入，RDB文件可以通过两个命令来生成 save

    1. SAVE：阻塞Redis的服务器进程，直到RDB文件被创建完毕
    2. BGSAVE：Fork（派生）出一个子进程来创建RDB文件，不阻塞服务器进程，父进程继续处理接收到的命令，子进程完成后会发送信号给父进程。可以通过lastsave指令来查看上次保存的时间

  • 自动化触发RDB持久化的方式

    1. 根据redis.conf配置里的SAVE m n 定时触发（用的是BGSAVE）
    2. 主从复制时，主节点发送RDB文件给从节点完成复制操作，主节点自动触发
    3. 执行Debug reload的时候
    4. 执行ShutDown且没有开启AOF持久化

  • 缺点：内存数据的全量同步，数据量大会由于IO而严重影响性能。

  • 可能因为Redis挂掉而丢失从当前至最近一次快照期间的数据

Redis做AOF持久化

  • AOF持久化是通过保存redis服务器的写状态来保存数据的，RDB持久化相当于备份数据状态，而AOF持久化是备份数据库接收到的指令，所有被写入到AOF的指令都是以Redis的协议格式来保存的
  • 数据库会记录下所有会变更数据库数据的命令，以append的形式追加保存到AOF文件中（增量的）
  • 设置appendonly yes 打开AOF持久化
  • 设置aof备份文件的名字
    1. always 表示一旦缓存区的内容变化，就总是及时的将缓存区的内容写到aof中
    2. everysec 表示每隔1s将缓存区的文件写入aof中
    3. no 表示将写入aof的操作交与操作系统来决定，一般而言，操作系统会等缓冲区填满才写到aof中
  • Redis支持在后台重建AOF文件 使用aof-rewrite

AOF和RDB的优缺点

  • RDB优点：全量数据快照，文件小，恢复快
  • RDB缺点：无法保存最近一次快照之后的数据
  • AOF优点：可读性高，适合保存增量数据，数据不易丢失
  • AOF缺点：文件体积大，恢复时间长

RDB-AOF混合持久化方式

  • BGSAVE做镜像全量持久化，AOF做增量持久化
  • 因为BGSAVE会耗费较长时间，不够实时，在停机的时候会导致大量丢失数据，需要AOF来配合使用，在Redis重启时，会使用BGSAVE持久化文件，重新构建内容，在使用AOF重放近期的操作指令。来实现完整恢复重启之前的状态

Pipeline 的好处

  • Pipeline和Linux的管道类似
  • Redis基于请求响应模型，单个请求需要一一应答
  • Pipeline批量执行指令，节省多次IO往返的时间
  • 有顺序依赖的指令建议分批发送

Redis的同步机制

  • 主从同步原理
  • Redis一般是一个master进行写的操作，若干个salver进行读的操作，master和salver代表一个个独立的redis server实例，定期的数据备份操作也是选择一个salver进行的，这样可以最大的发挥出redis的性能，保证redis 的数据的弱一致性，即最终一致性，不需要保证master和salver的数据是实时同步的，但是在过了一段时间后他们的数据是趋于同步的，这就是最终一致性。
  • Redis可以使用主从同步或者从从同步，第一次主节点做一次bgsave，并同时将后续修改操作记录到内存的buffer里去，待完成后将rdb文件全量同步到从节点里面，从节点接收完成后，就将rdb的镜像加载到内存中，加载完成后再通知主节点，将期间修改的操作数据以及增量数据同步到从节点。
  • 全同步过程
    • Slave发送sync命令到Master
    • Master启动一个后台进程，将Redis中的数据快照保存到文件中
    • Master将保存数据快照期间接收到的写命令缓存起来
    • Master完成写文件操作后，将该文件发送给Slave
    • Slave使用新的RDB文件替换掉旧的RDB文件
    • Master将这期间收集的增量写命令发送给Slave端
  • 增量同步过程
    • Master接收到用户的操作指令，判断是否需要传播到Slave
    • 然后将操作记录追加到AOF文件中
    • 将操作传播到其他Slave
      • 对齐主从库
      • 往响应缓存写入指令
    • 将缓存中的数据发送给Slave
  • 弊端：
    • 当master挂掉之后就不能提供写服务了

Redis Sentinel

哨兵 解决主从同步Master宕机后的主从切换问题:

  • 监控：检查主从服务器是否允许正常
  • 提醒：通过API向管理员或者其他应用程序发送故障通知
  • 自动故障迁移，主从切换，会将失效的主服务器的一个从服务器升级为主服务器，并让之前的Slaver改为复制新的Master。当客户端试图连接失效的主服务器时，集群也会向客户端返回新Master的地址，使得集群可以使用新的master代替失效的master，
  • Redis Sentinel 是一个分布式系统，你可以再同一个架构中允许多个Sentinel进程，这些进程通过使用流言协议来接收主服务器是否下线的信息，并使用投票协议来决定是否执行自动故障迁移，以及选择哪一个Salver来作为新的Master

session和application 的区别

  1. 作用域不同
    • session是用户级别的，application是web应用级别的，是一个全局作用域
    • 一个用户一个session ，每一个用户的session的各不相同。
    • 一个web应用一个application
  2. 生命周期不同
    • session 在登陆后创建，在用户注销，关闭浏览器的时候销毁
    • application 在创建web应用程序的时候创建，在关闭应用是销毁

Cookie 和Session的区别

  1. cookie存储在客户端，不安全，Session存在服务器，安全

  2. cookie 可以存放用户名等不重要的信息，可以减缓服务器的压力，session 存储比较重要的信息，session 会存在 服务器上，会占用内存

  3. 单个cookie保存的数据不能超过4k，很多浏览器最多保存20个cookie
