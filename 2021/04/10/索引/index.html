    id 表示sql语句的执行顺序，比如说复合查询 里面的语句是先执行的

    type

    如果type 为index 和all 的话 表示的是全表扫描。如果该语句是查询的话，type 又是这两个的话，就得优化一下了

    Extra

    ​ 如果extra 中出现以下两项意味着MySQL 根本不能使用索引，效率会受到重大影响

    1. Using filesort 表示 Mysql 会对结果使用一个外部索引排序，而不是从表里按索引次序读到相关内容。可能在内存或者磁盘上进行排序。 Mysql中无法利用索引完成的排序操作称为 文件排序
    2. Using temporary 表示Mysql 在对查询结果排序时使用临时表。常见于排序order by 和分组查询 group by

    看刚才的explain type 为 all extra 为 using filesort 这就很需要调优了

联合索引的最左匹配原则的成因

假设有两列 A，B，我们给A，B设置为索引，顺序是A，B，在where语句中调用 A = 多少，B = 多少的时候就会走A，B联合索引。如果我们调用 where A = 多少的时候也会走这个索引，但是我们调用B= 什么的时候他就不走这个索引了

  1. 最左前缀匹配原则：mysql会一直向右匹配知道遇到范围查询（>,<,between,like）就停止匹配，比如a = 3 and b = 4，and c > 5 and d = 6 如果建立（abcd）顺序的索引，d是用不到索引的，如果建立 （abdc）的索引则都可以用到，abd的顺序可以任意调整
  2. = 和 in 可以乱序，比如 a = 1 and b= 2 and c = 3 建立abc索引可以任意顺序，mysql的查询优化器会帮你一优化成索引可以识别的形式

最左匹配原则的成因：

mysql 创建联合索引是根据最左边也就是第一个字段进行排序，在第一个字段排序基础上，在对第二个字段进行排序（就类似于order by 字段1 再order by 字段2的一种排序规则），所以第一个字段是绝对有序，而第二个字段就是无序的了，因此直接使用第二个字段进行where查询是用不到第二个索引的

索引顺序是col3 col2 ，先按照col3给索引创建b+tree ，然后在根据col2查找，如果直接使用col2的话，是不能走这个b+tree的

索引是建立的越多越好嘛？

  • 数据量小的表不需要建立索引，建立会增加额外的索引开销
  • 数据变更需要维护索引，因此更多的索引意味着更多的维护成本
  • 更多的索引也需要更多的存储空间

MyISAM 与 InnoDB 关于锁方面的区别是什么？

  • MyISAM 默认用的是表级锁，不支持行级锁

  • InnoDB默认使用的是行级锁，也支持表级锁

    • 对于myisam 当对数据进行查询的时候，会自动添加一个表级别的读锁，当对数据进行增删改的时候，会添加一个写锁。 读锁未被释放的时候，另一个session对同一张表添加一个写锁的时候，就会被阻塞。直到所有的读锁都被释放为止
