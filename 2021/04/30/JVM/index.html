平台无关是如何实现的

java源码首先被编译成字节码，java提供了各种不同平台上的虚拟机,再由不同平台的JVM进行解析，Java语言在不同平台上运行时不需要进行重新编译，Java虚拟机在执行字节码的时候，把字节码转换成具体平台上的机器指令。

为什么JVM不直接将源码解析成机器码去执行呢？

如果我们直接将源码解析成机器码去执行，每次执行都需要进行语法的检查，这样整体的性能受到影响，引入中间字节码后，能够保证在被编译成字节码之后，多次执行程序是不需要执行各种校验的。2. 同时可以脱离java的束缚，有Gove，Salc来生成字节码，同样可以被jvm进行调用执行，进而也能增加平台的兼容扩展能力

JVM如何加载.class文件

Java虚拟机：是一种抽象化的计算机，JVM有自己完善的硬件架构，如处理器，堆栈，寄存器等，还具有相应的指令系统。JVM屏蔽了与具体平台相关的信息，使得Java程序只需在java虚拟机上运行字节码就可以在多个平台运行代码。屏蔽不通过操作系统的不同并且减少基于原始语言的复杂性

ClassLoader的作用：加载class文件到内存中 ，具体能不能执行得看Execution Engine

Execution Engine 解释器：负责对命令进行解析，解析完成后，就交给操作系统去执行

Native Interface ：融合不同开发语言的原生库为java所用，具体是 Native Method Stack 登记native 方法，在执行时加载Native Libraies

Runtime Data Area : 就是内存数据模型，我们所写的程序都会被加载到这里，之后才会被运行

JVM主要由Class Loader Runtime Data Area ，Execution Engine ， Native Interface 这四个部分组成，主要通过class loader ，将符合其格式要求的class文件加载到内存里，并通过Execution Engine 解析class文件里面的字节码，提交给操作系统去执行

类从编译到执行的过程

  1. 编译器将Robot.java源文件编译为Robot.class字节码文件
  2. ClassLoader将字节码转换为JVM中的Class对象
  3. JVM利用Class对象实例化Robot对象

谈谈ClassLoader

  • 概述

    ClassLoader 主要工作在Class加载阶段，将Class文件里的二进制数据流转载进系统，然后交给Java虚拟机进行连接，初始化等操作

  • ClassLoader的种类

    1. BootStrapClassLoader :C++编写，加载核心库java.*
    2. ExtClassLoader: java编写，加载扩展库javax.*
    3. AppClassLoader: java编写，加载程序所在目录，环境变量classpath

  • 自定义ClassLoader: Java编写，定制化加载,可能不在系统的classpath的范围内，可以不是class文件或jar文件。有很多用途

    findClass 就是根据名称或位置来加载字节码，会调用defineClass来解析定义字节流，返回Class对象。

    通过自定义的ClassLoader，我们不仅可以通过自定义路径来加载class文件，还可以从网上获取二进制的流，来生成二进制的类，还可以对某些敏感的class文件进行加密，在findclass里面对其进行解密，也可以修改二进制的流，给类添加信息(ASM字节码生成技术。也可以做为AOP的联想)

  • ClassLoader的双亲委派机制

    从前面了解到，不同的classLoader加载类的方式和路径有所不同，为了分工，各自负责各自的区块，各司其职，所以存在一个机制，让他们相互写作，形成一个整体，这就是双亲委派机制

    S首先自底而上的检查类是否已经加载，从自定义的类加载器开始找有没有加载过该类，如果加载过直接返回，如果没有，就委派给他的parent也就是App ClassLoader，如果也没有，就找Extension ClassLoader，如果没有，就到最顶层Bootstrap ClassLoader。如果也没有，就去rt.jar或者-Xbootclasspath指定的jar包。如果还没有，就委托给Extension ClassLoader 去ext目录下的jar文件中找，或者去-Djava.ext.dirs指定目录下的jar包.

  • 双亲委派机制可以防止多份同样字节码的加载，如果不适用，每个类都各自加载字节码文件，内存中就会有重复的字节码文件存在,使用双亲委派机制，只会加载一次该类

类的加载方式

  • 隐式加载 new 可以使用构造方法给类初始化值
  • 显示加载: loadclass ，forName ，使用newInstance 创建对象，通过反射给类赋值。

loadClass和forName的区别

  • 都能获得任意一个类的所有方法和熟悉
  • loadClass的没有链接过程（resolve=false），而forName有链接过程（initial = true）
  • Class.forName得到的class是已经初始化完成的 会执行static
  • Class.loadClass得到的class是还没有链接的 不会执行 static
  • forName的使用场景: jdbc.Driver 中有static代码
  • loadClass的是使用场景: SpringIoC 使用的懒加载技术，由于loadClass不用初始化，所以使用loadClass来加快懒加载速度，把类的初始化工作留到类的使用的时候在去做

类的装载过程

  • 加载
    • 通过ClassLoader的loadClass方法将字节码文件加载到内存中，并将这些静态数据转换成运行时数据区的方法区的类型数据，在运行时，数据区堆中生成一个代表这个类的java.lang.Class对象，作为方法区类的访问入口
  • 链接 (非必须)
    • 校验 检查加载的class文件的正确性和安全性
    • 准备 为类变量分配存储空间，为类变量设置初始值，类变量随类型信息存放在方法区中，生命周期很长，使用不当可能造成内存泄漏 类变量就是static信息。初始值并不是实际要赋的值
    • 解析 JVM将常量池内的符号引用转换为直接引用
  • 初始化
    • 执行类变量赋值和静态代码块

JVM的内存模型

  • 程序计数器:

    • 是一块较小的内存空间,当前线程所执行字节码的行号指示器，在虚拟机的概念模型里，字节码解释器工作时，就是

  • 改变计数器的值来选取下一条需要执行的字节码指令，包括分支，循环，跳转，异常处理,线程恢复等基础功能，都需要依赖这个计数器来完成

    • 在任何一个确定的时刻，一个处理器只会执行一条线程中的指令，因此，为了线程切换后能回复到正确的执行位置，每条线程都需要独立的程序计数器，各条线程之间的计数器互不影响，独立存储，我们称类内存区域为线程私有的内存

  • 对java方法计数，如果是Native方法则计数器值为Undefined

    • 由于只是记录行号，不会发生内存泄漏的问题

  • 程序计数器是逻辑计数器，而非物理计数器，为了线程切换后都能回到原来的位置，每个线程都有一个独立的程序计数器，是线程独立的，并且只为Java方法计数，native方法对应的计数器是undefin，使用程序计数器不用担心发生内存泄漏的问题

  • Java虚拟机栈

    • Java方法指向的内存模型

    • 每个方法被执行时都会创建一个栈帧，即方法运行时的基础数据结构，用来存储局部变量表，操作栈，动态连接，返回地址

    • 每个方法执行中，对应虚拟机栈帧，从入栈到出栈的过程，java虚拟机栈用来存储栈帧，而栈帧持有局部变量，和部分结果，与参与方法的部分返回，当方法调用结束时，帧才会被销毁。

    • 虚拟机栈包含了单个线程每个方法执行的栈帧，栈帧则存储了局部变量表，操作数栈，动态连接，和方法出口，等信息

      • 局部变量表：包含了方法执行过程中的所有变量，包括this引用，所有方法参数，其他局部变量，包括布尔值，byte，char，long short int double 等。

      • 操作数栈：在执行字节码指令过程中会用到，这种方式类似于原生cpu寄存器，大部分jvm字节码把时间花费在操作数栈的操作上，包括入栈，出栈，复制，交换，产生消费变量的操作，因此，局部变量数组和操作数栈之间的交换变量指令操作通过字节码频繁执行

        我们都知道栈是一个后进先出的数据结构，因此当前执行的方法在栈的顶部，每次方法调用时，一个新的栈帧创建，并压栈到栈顶，当方法正常返回或抛出未捕获的异常时，栈帧就会出栈，除了栈帧的压栈和出栈，栈不能被直接操作。局部变量表给操作数栈做一个数据支撑

  • Java递归为什么会造成stackoverflow 异常

    • 当线程执行一个方法时，就会随之创建一个栈帧，并将加入的栈帧加入到虚拟机栈中，当方法执行完毕的时候，便会对栈帧出栈，因此可知，线程当前执行的方法，所对应的栈帧，必定位于java栈的顶部，而我们函数不断调用自身，每次调用方法都会创建栈帧，保存栈帧状态，将他放在虚拟机栈中，栈帧切换时会切换到最新的方法栈帧中，而我们每个线程的虚拟机栈的深度固定的，递归实现会导致栈深度的增加，每次递归都会往栈里压一个栈帧，如果这一消耗超过了栈允许的最大深度，就会报stackOverFlowError异常，解决这个问题的思路就是限制递归的深度，或者将递归变成迭代

    虚拟机栈也是java虚拟机自动管理的，栈类似于一个集合，但是他是有固定的容量的，是由多个栈帧合起来的，再我们编写代码的时候，当我们允行程序的时候，每调用一个方法，java虚拟机就会分配对应的一块空间，这个空间就是栈帧啦，而当方法调用结束后，对应的栈帧就会自动释放掉，这也是栈的空间不需要gc，而会自动回收的原因啦

    元空间（metaSpace）与永久代（PermGen）的区别

再jdk8之后，将类的元数据放到本地堆内存中，这一块就叫metaSpace，该区域在jdk7及以前是属于永久代的，元空间和永久代都是用来存储class对象的相关信息，包括method 和field等，二者都是方法区的实现，只是实现有所不同，所以说方法区只是一种jvm的规范，在java7之后，原先位于方法区的字符串常量池已被移动到了java堆中，并且在jdk8，元空间替代了永久代。

  • 元空间使用本地内存，而永久代使用jvm内存
  • 理论上本地内存还有多少，meteSpace就能有多大，解决了空间不足的问题，jvm默认会根据运行时的需要动态设置其大小.

java堆

  • java堆是jvm管理的最大的内存，是被所有线程共享的内存区域，对象实例的分配区域，存放对象实例
  • GC管理的主要区域，可以分为新生代，老年代
