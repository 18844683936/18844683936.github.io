单例模式

发表于2022-04-04|更新于2022-12-15
|阅读量:

定义与类型

  1. 定义: 保证一个类仅有一个实例，并提供一个全局访问点
  2. 类型：创建型

使用场景

  1. 想确保任何情况下都绝对只有一个实例
    1. 例如单服务的情况下，计数器可以使用单例，集群服务下，就得用共享的
    2. 应用配置，在使用线程池的时候，一般也是单例
    3. 数据库的连接池，一般也是单例的

优点

  1. 在内存里只有一个实例，减少了开销，特别是在一个对象需要频繁创建销毁时，而且创建和销毁这个过程还不能优化，这个时候，单例模式的优点就显示出来了
  2. 可以避免对资源的多重占用， 例如我们给一个实例进行写操作，由于只有一个实例存在内存中，可以避免对同一个资源文件同时写操作
  3. 设置了全局访问点，严格控制访问，也就是对外我不让你 new出来， 你只能通过我这个方法来创建对象

缺点

  1. 没有接口，扩展困难，如果想扩展的话，基本上都要修改代码

重点

  1. 私有构造器 ：禁止单例类从外部调用构造函数来创建这个对象，为了达到这个目的，必须设置类的构造函数为private

  2. 线程安全

  3. 延迟加载 ： 使用它的时候再创建

  4. 序列化和反序列化

  5. 反射

  3. 假设线程0重排序了，线程0先设置了instance指向了内存空间，线程1开始自上而下判断instance是否为null，这时候判断出并不为null，然后线程1开始访问对象，也就是说线程1比线程0更早的访问对象，所以说，线程1访问到的对象是还没有初始化的对象，这时候就有问题了，系统就会报异常了

  4. 解决方法

    • 不允许线程0重排序 使用 volatile 来修饰instance 来实现线程安全的初始化，重排序就会被禁止，在多线程的时候cpu也有共享内存，加了volatile关键字后，所有的线程都可以看到共享内存的最新状态，保证了内存的可见性，在用volatile修饰的变量在进行写操作的时候，会多出一些汇编代码，起到两个作用，第一，会将当前处理器缓存行的数据写回到内存中，这个写回内存的操作，会将其他cpu中缓存了该内存地址的数据无效，因为他们的数据无效了，所以他们又从共享内存中同步数据，这样就保证了内存的可见性，这里主要是使用缓存一致性协议，当处理器发现，我这个内存已经无效了，会重新将数据从共享内存中读到处理器的缓存中
    • 允许线程0重排序，不允许线程1看到线程0的重排序

  5. ```java
    public class LazyDoubleCheckSingleton {

    private static volatile LazyDoubleCheckSingleton lazyDoubleCheckSingleton = null;
private LazyDoubleCheckSingleton(){
}

public static LazyDoubleCheckSingleton getInstance(){
      /*
        * 我们进行了两次 if (singleton == null) 检查，这就是“双重检查锁”这个名字的由来。
        * 这种写法是可以保证线程安全的，假设有两个线程同时到达 synchronized 语句块，那么实例化代码只会由其中先抢到锁的线程执行一次，
        * 而后抢到锁的线程会在第二个 if 判断中发现 singleton 不为 null，所以跳过创建实例的语句。再后面的其他线程再来调用 getInstance 方法时，
        * 只需判断第一次的 if (singleton == null) ，然后会跳过整个 if 块，直接 return 实例化后的对象。
        这种写法的优点是不仅线程安全，而且延迟加载、效率也更高。
        * */
    
       /*
            虽然判断了是否为空，但是有一种情况是，虽然他不为空，但是他并没有初始化完成，
            如果发生这种情况，就会返回一个未初始化完成的单例类，导致之后这个类的调用出现问题，
            使用volatile关键字可以保证即使你现在返回的是未初始化完成的单例类，
            但是当其他线程初始化完成之后，你之前得到的未初始化的类也会更新为初始化完成的单例类
            */
    if (lazyDoubleCheckSingleton == null){
        synchronized (LazyDoubleCheckSingleton.class){
            if (lazyDoubleCheckSingleton==null){
                lazyDoubleCheckSingleton = new LazyDoubleCheckSingleton();
                /*上面这一行看上去只是一行，实际上是三个步骤
                1. 分配内存给这个对象
                2. 初始化对象
                3. 设置lazyDoubleCheckSingleton 指向分配的内存地址
                2 和 3 的顺序有可能被颠倒
                intra-thread semantics 例如2，3的顺序被颠倒了也保证不会改变单线程内的执行结果
                解决方法:使用 volatile 关键字来修饰lazyDoubleCheckSingleton 防止重排序
                */
            }
        }
    }
    return lazyDoubleCheckSingleton;
}

    }

  7. JVM在类初始化阶段，也就是class被加载后，并且被线程使用之前，都是类的初始化阶段，在这个阶段会执行类的初始化，在此期间，JVM会获取一个类的锁，这个锁会同步多个线程对一个类的初始化，也就是上图绿色部分，基于这个特性，我们可以实现基于静态内部类的并且线程安全的延迟初始化方案，这样线程1并不会看到重排序的过程

  8. 初始化一个类，包括执行这个类的静态初始化，初始化在这个类中声明的静态变量，主要是分为五种情况，首次发生时，一个类会被立刻初始化（包括interface）

    • 有一个A类的实例被创建
    • A类中声明的一个静态方法被调用
    • A类中声明的静态成员被赋值
    • A类中声明的静态成员被使用，并且这个成员不是一个产量成员
    • A类如果是一个顶级类，并且在A类中有嵌套的断言语句，这种情况A类也会被立刻初始化

  9. 当线程0，线程1要获得初始化锁的时候，这个时候肯定只有一个能获得这个锁，假设线程0获得这个锁，线程0执行这个类的初始化过程，对于静态内部类，即使2，3之间存在重排序，但是线程1是无法看到这个重排序的 ，因为这里有个Class对象的初始化锁。

  2. 来看看源码，为啥枚举类的序列化反序列化得到的是同一个对象？ 首先获得枚举对象的名称，然后将result声明成null，再2012行进行赋值，通过类型和name获得枚举常量，由于枚举中的name是唯一的，并且对应一个枚举常量，所以2011行获得的是唯一的枚举常量，这样就没有创建对象，维持了单例属性

容器单例模式

  1. ```java
    public class ContainerSingleton {

    //很明显，使用HashMap 线程肯定是不安全的，如果我们在类初始化的时候，就把这个map初始化完成，就是说把所有的单例对象都生成完，
// 放到这里边，用的时候直接用，这样也是可以的
//考虑到一种情况，就是初始化的时候并发的向Controlmap中放入对象，如果key都相同的话，
// 有可能向map中存多次相同key的对象，但是初始化完成后，取对象的时候用的是同一个对象
// 也有可能是 存了对象后直接取用，然后第二次存了对象后再次获取，这样获取到的对象就不是同一个对象了
// 使用HashTable或则ConcurrentHashMap来添加锁，
// 但是ConcurrentHashMap，我们使用了静态的map，而且还直接操作了这个map，所以这个map也并不是绝对的线程安全

//所以：如果系统中，单例对象非常多的时候，也可以考虑用一个对象将所有单例对象同一管理，优点是统一管理，节省资源，map相当于一个缓存，缺点是，线程不安全
private static Map<String,Object> singletonMap = new ConcurrentHashMap<>();

private static void  putInstance(String key,Object instance){
   if (Objects.nonNull(instance) && Strings.isNotBlank(key)){
       if (!singletonMap.containsKey(key)){
           singletonMap.put(key,instance);
       }
   }
}

public static Object getInstance(String key){
    return singletonMap.get(key);
}

private ContainerSingleton(){
    
}

    }

  3. ThreadLocal 会为每一个线程提供独立的变量副本，ThreadLocal是基于ThreadLocalMap来实现的，调用ThreadLocal的get方法时，走的就是这个map，不用指定key,他维持了线程之间的隔离，ThreadLocal隔离了多个线程对数据的访问冲突，对于多线程资源共享的问题，如果使用同步锁，就是以时间换空间的方法，如果用threadlocal就是空间换时间的方式，他会创建很多对象，一个线程里至少创建一个对象，但是对于这个线程，他获取对象是唯一的
