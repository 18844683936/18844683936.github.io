桥接模式

发表于2022-04-26|更新于2022-12-15
|阅读量:

定义与类型

  1. 将抽象部分与它的具体实现部分分离，使他们都可以独立地变化，
  2. 通过组合的方式建立两个类之间的联系，而不是继承（合成复用原则推荐优先使用组合而不是继承）
  3. 类型：结构型
  4. 可以防止子类过多，类爆炸这种情况

适用场景

  1. 抽象和具体之间增加更多的灵活性，适用桥接模式可以避免在这两个层次之间使用静态的继承关系，通过桥接使他们之间建立一个关联关系，并且抽象部分和实现部分都可以独立的扩展而且互不影响，就可以动态的将一个抽象化对象和一个实例化对象进行组合，也就实现了他们之间的解耦
  2. 一个类存在两个（或多个）独立变化的维度，且这两个（多个）维度都需要独立进行扩展，抽象的部分可以独立的扩展，具体的部分也可以独立的扩展
  3. 不希望使用继承，或因为多层继承导致系统类的个数剧增

优点

  1. 分离抽象部分以及其具体实现部分
  2. 提高了系统的可扩展性
  3. 符合开闭原则和合成复用原则

缺点

  1. 增加了系统的理解与设计难度，要求我们在编码的时候，一开始就对抽象层进行设计和编程
  2. 需要正确的识别出系统中两个独立变化的维度

相关设计模式

  1. 桥接模式和组合模式，组合模式更强调的是部分和整体之间的组合，而桥接模式强调的是平行级别下不同类的组合
  2. 桥接模式和适配器模式，这两个模式都是为了两个东西互相配合工作，适配器模式是改变已有的接口，让他们之间可以相互配合，让功能相似，但是接口不同的类适配起来，而桥接模式是分离抽象和具体的实现，目的就是分离，。

Coding

  1. 场景：中国有很多银行，有农行，工行，我们有账号，分为定期账号和活期账号，定期账号可以存半年，一年，银行给与固定的利息，这里分两块，一块是银行，一块是账号

  3. 这个模式可以不限制的增加Account和Bank的子类，而且还不会造成类爆炸

  4. /**
 * @description: 这里面有两个方法，一个是打开我们的账号，分为打开哪个银行的账号，另一个是查看账号的类型，是活期储蓄还是定期储蓄，
 * @author: <a href="mailto:guang.chen@zkteco.com">guang.chen</a>
 * @create: 2022-04-26
 */
public interface Account {
    Account openAccount();
    void showAccountType();
    //我们有中国农业银行，中国工商银行  还有活期账号，定期账号，交叉一下就会有四种组合， 一旦说银行不断的扩展，账号类型也不断的扩展的话，、
    //那么对于桥接模式就在适合不过了，他们两个都可以随着自己的层级进行扩展
}


/**
 * @description: 定期账户
 * @author: <a href="mailto:guang.chen@zkteco.com">guang.chen</a>
 * @create: 2022-04-26
 */
public class DepositAccount implements Account{
    @Override
    public Account openAccount() {
        System.out.println("打开定期账号");
        return new DepositAccount();
    }

    @Override
    public void showAccountType() {
        System.out.println("这是一个定期账号");
    }
}


/**
 * @description:
 * @author: <a href="mailto:guang.chen@zkteco.com">guang.chen</a>
 * @create: 2022-04-26
 */
public class SavingAccount implements Account{
    @Override
    public Account openAccount() {
        System.out.println("打开活期账号");
        return new SavingAccount();
    }

    @Override
    public void showAccountType() {
        System.out.println("这是一个活期账号");
    }
}

public abstract class Bank {
    //为啥是抽象类，因为我们要把Account引入到Bank里面，通过这种组合的方式，交给子类来处理这种行为，
    // 而这个行为就是openAccount，至少我要确定，我打开的是哪个银行的账号，

    protected Account account;
    public Bank (Account account){
        this.account = account;
    }

    //和Bank中声明和Account中相同的方法并不是强制的，
    // 是为了更方便的理解，
    // Bank中的方法要委托给Account中的openAccount方法，所以就声明成了一样的方法名
    abstract Account openAccount();


}

public class ABCBank extends Bank {
    public ABCBank(Account account) {
        super(account);
    }

    @Override
    Account openAccount() {
        System.out.println("打开中国农业银行账号");
        //不要把所以的实现都自己完成，而是委托给account来完成，否则就没有意义了
        // 无论Account如何扩展，这个类是不用变化的
        account.openAccount();
        return account;
    }
}

public class ICBCBank extends Bank {
    public ICBCBank(Account account) {
        super(account);
    }

    @Override
    Account openAccount() {
        System.out.println("打开中国工商银行账户");
        account.openAccount();
        return account;
    }

}


public class Test {
    public static void main(String[] args) {
        ABCBank abcBank = new ABCBank(new SavingAccount());
        abcBank.openAccount().showAccountType();
    }
}
