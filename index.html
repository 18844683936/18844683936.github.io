Spring事务

事务传播 - Propagation

  1. REQUIRED（默认的，多用于增删改）：使用当前的事务，如果当前没有事务，则自己新建一个事务，子方法是必须运行在一个事务中的；如果当前存在事务，则加入这个事务，成为一个整体
  2. SUPPORTS（多用于查询）: 如果当前有事务，则使用事务，如果当前没有事务，则不使用事务
  3. MANDATORY: 强调必须存在一个事务，谁调用我，谁就必须要有一个事务，如果他没有事务，我就抛出一个异常
  4. REQUIRES_NEW: 如果当前没有事务，则创建一个新的事务（同REQUIRED），如果当前有事务的话，就挂起当前的事务。两个事务是分开的，如果父方法报错，子方法不会收到影响，因为子方法使用了一个新的事务
  5. NOT_SUPPORTS: 如果当前有事务，则把事务挂起，自己不使用事务，无论如何自己都不使用事务
  6. NEVER: 如果当前有事务，则抛出异常
  7. NESTED：如果当前有事务，则开启子事务（嵌套事务），嵌套事务是独立提交或者回滚;如果当前没有事务，则同REQUIRED，
    但是如果主事务回滚，则子事务会一起回滚。相反，子事务异常，则父事务可以回滚或不回滚

桥接模式

2022-04-26

定义与类型

  1. 将抽象部分与它的具体实现部分分离，使他们都可以独立地变化，
  2. 通过组合的方式建立两个类之间的联系，而不是继承（合成复用原则推荐优先使用组合而不是继承）
  3. 类型：结构型
  4. 可以防止子类过多，类爆炸这种情况

适用场景

  1. 抽象和具体之间增加更多的灵活性，适用桥接模式可以避免在这两个层次之间使用静态的继承关系，通过桥接使他们之间建立一个关联关系，并且抽象部分和实现部分都可以独立的扩展而且互不影响，就可以动态的将一个抽象化对象和一个实例化对象进行组合，也就实现了他们之间的解耦
  2. 一个类存在两个（或多个）独立变化的维度，且这两个（多个）维度都需要独立进行扩展，抽象的部分可以独立的扩展，具体的部分也可以独立的扩展
  3. 不希望使用继承，或因为多层继承导致系统类的个数剧增

优点

  1. 分离抽象部分以及其具体实现部分
  2. 提高了系统的可扩展性
  3. 符合开闭原则和合成复用原则

缺点

  1. 增加了系统的理解与设计难度，要求我们在编码的时候，一开始就对抽象层进行设计和编程
  2. 需要正确的识别出系统中两个独立变化的维度

相关设计模式

  1. 桥接模式和组合模式，组合模式更强调的是部分和整体之间的组合，而桥接模式强调的是平行级别下不同类的组合
  2. 桥接模式和适配器模式，这两个模式都是为了两个东西互相配合工作，适配器模式是改变已有的接口，让他们之间可以相互配合，让功能相似，但是接口不同的类适配起来，而桥接模式是分离抽象和具体的实现，目的就是分离，。

组合模式

2022-04-20

定义与类型

  1. 定义：将对象组合成树形结构以表示 “部分-整体” 的层次结构
  2. 组合模式使客户端对单个对象和组合对象保持一致的方式处理
  3. 类型：结构型
  4. 例如：我们有一个树形结构的菜单，在菜单里面还有子菜单，在子菜单中可能还有子菜单的子菜单，在这种树形结构下，我们就可以使用组合模式。

适用场景

  1. 希望客户端可以忽略组合对象与单个对象的差异时
  2. 处理一个树形结构时

优点

  1. 清楚的定义分层次的复杂对象，表示对象的全部或部分层次
  2. 让客户端忽略了层次的差异，方便对整个层次结构进行控制
  3. 简化客户端代码
  4. 符合开闭原则

缺点

  1. 限制类型时会较为复杂
  2. 使设计变得更加抽象

相关设计模式

  1. 组合模式和访问者模式，适用访问者模式来访问组合模式中的递归调用

Coding

享元模式

2022-04-20

定义与类型

  1. 定义：提供了减少对象数量从而改善应用所需的对象结构的方式
  2. 运用共享技术有i下地支持大量细粒度的对象
  3. 结构型
  4. 减少对象的数量，减少内存的占用，从而提高性能

适用场景

  1. 常常应用于系统底层的开发，以便解决系统的性能问题
  2. java中的String类型就使用了享元模式，如果有则返回，如果没有，就创建一个字符串放到字符串缓存池中
  3. 数据库的连接池，也都是创建好的连接对象，需要的时候拿来用，不需要的时候就放回去
  4. 在系统中有大量对象时，就有可能造成内存溢出，我们把其中共同的部分抽象出来，如果有相同业务请求，就把原有的对象返回给他
  5. 系统有大量相似对象，需要缓冲池的场景
  6. 一个系统中有大量的细粒度对象，这些细粒度对象 的 状态大部分都可以外部化

优点

  1. 减少对象的创建，降低内存中对象的数量，降低系统的内存，提高效率
  2. 减少内存之外的其他资源占用 ，比如说时间，我们如果平常去new 一个对象要花费时间，但是适用这个设计模式，会减少new的次数，从而减少时间，还有操作系统中文件句柄也都是一种资源

缺点

  1. 关注内/外部状态，关注线程安全问题
  2. 系统程序的逻辑复杂化

扩展

  1. 内部状态 在享元对象的内部，不会随着环境的变化而变化的部分
  2. 外部状态 随着环境改变而改变的状态 不可以共享的状态，这个状态时记录在享元对象的外部的

相关的设计模式

  1. 享元模式和代理模式 : 代理模式肯定是要代理一个类，如果生成这个代理类需要花费的资源和时间都比较多，那我们就可以使用享元模式来提高程序的处理速度
  2. 享元模式和单例模式：容器单例

适配器模式

2022-04-18

定义与类型

  1. 将一个类的接口转换成客户期望的另一个接口
  2. 使原本接口不兼容的类可以一起工作
  3. 类型：结构型

适配器应用场景

  1. 已经存在的类，他的方法和需求不匹配时，方法结果相同或相似，适配器为了复用现有的类，系统的数据和行为都正确，但是接口不符合，这个时候就可以采取适配器模式，使原有接口匹配
  2. 不是软件设计阶段考虑的设计模式，是随着软件维护，由于不同产品，不同厂家造成功能类似而接口不相同的情况下的解决方案
  3. 想使用一个已经存在的类，当如果他的接口，也就是他的方法和你要求不相同时，就应该考虑适配器模式 ， 两个类所做的事情相同或相似，但是具有不同的接口时要使用他，客户代码可以统一调用同一接口

优点

  1. 能提高类的透明性和复用，现有的类复用但不需要改变，解决了现有类和目标类不匹配的问题
  2. 目标类和适配器类结构，提高程序扩展性
  3. 符合开闭原则，具体的实现都在适配器中，客户端知道的只有适配器类，扩展的话，也只需要扩展适配器类就可以

缺点

  1. 编写过程需要全面考虑，可能会增加系统的复杂性
  2. 增加系统代码可读的难度，如果过多的使用适配器，会使系统凌乱，不易整理，

扩展

  1. 对象适配器
    • 符合组合复用原则，并且使用委托机制
  2. 类适配器
    • 使用类继承来实现的

相关设计模式

  1. 适配器模式和外观模式
    • 都是对现有的类，先存的系统的封装，外观定义了新的接口，而适配器则是复用一个原有的接口，适配器使两个已有的接口协同工作，而外观在现有的系统中提供一个更为方便的访问入口，如果强行的把外观模式称为适配的话，那这两个最大的区别在于适配的力度不同，外观模式是用来适配整个子系统，力度更大

装饰者模式

2022-04-12

定义与类型

  1. 定义:在不改变原有对象的基础之上，将功能附加到对象上，提供了比继承更有弹性的替代方案（扩展原有对象功能）
  2. 类型：结构性
  3. 比如说，想买个蛋糕，想加个草莓，再加个芒果，根据不同的需求来装饰这个蛋糕，房子装修也是一样的，还有礼品包装选择不同的包装盒

适用场景

  1. 扩展一个类的功能或给一个类添加附加职责
  2. 动态的给一个对象添加功能，这些功能可以再动态的撤销

优点

  1. 继承的有力补充，比继承灵活，不改变原有对象的情况下给一个对象扩展功能
  2. 平常我们使用集成来扩展类的功能，但是如果要扩展的功能的种类繁多，势必会生成很多子类来增加系统的复杂性，同时使用集成来实现扩展功能我们必须可预见这些功能，因为这些功能在编译时就确定了，是静态的，如果使用装饰者模式，这些功能是由用户或应用层的代码来动态的决定加入的方式和时间，装饰者模式提供了一种即插即用的方法，在运行期间决定何时增加何种功能，
  3. 装饰者模式本身也是建立在集成的功能能上的
  4. 通过使用不同装饰类以及这些装饰类 的排列组合，可以实现不同效果
  5. 符合开闭原则，使用装饰者模式，我们可以扩展行为，并且具体的被装饰者和具体的装饰者可以独立变化，原有的代码不用改变，符合开闭原则，装饰者做的，是将类中的装饰功能，从类中移除出去，这样简化了原来被装饰的类，同时把类的核心职责和装饰功能区分开，这样还能去除相关类中重复的装饰逻辑，

缺点

  1. 会出现更多的代码，更多的类，增加程序的复杂性，因为要抽象，并且创建很多装饰者的类
  2. 动态装饰式，多层装饰时会更复杂，比如说，我们的程序呢，根据用户传入的参数，来决定动态的装饰哪些类，装饰者使用的类比继承要少，但是比继承关系使用更多的对象，更多的对象在排查错误的时候会变得困难，特别是这些对象看上去又非常像，因为装饰者会继承被装饰对象，而被装饰对象又具有具体的实体，他们在父类看起类是同一类对象，所以会增加一定复杂度。

相关设计模式

  1. 装饰者模式和代理模式，装饰者模式关注在一个对象上动态的添加方法 ，而代理模式关注在控制对对象的访问，代理模式的代理类可以对他的客户隐藏一个对象的具体信息，通常使用代理模式的时候，常常在代理类中创建一个对象的实例，当我们使用装饰者模式的时候，我们通常会把原始对象作为一个参数，传给装饰者的构造器，这个是使用上的不同
  2. 装饰者模式和适配器模式，都可以叫做包装者模式，wrapper，装饰者和被装饰者可以实现相同的接口，或者装饰者是被装饰者的子类，适配器模式中，适配器和被适配的类具有不同的接口，也有可能有部分的接口是重合的，装饰者模式还可以退化为半装饰者，也即是一个装饰者除了提供被装饰类的接口外呢，还提供了其他的方法，就变成了一个半透明的装饰者，如果我们应用层代码想使用这个提供的特殊的方法时，就需要具体的装饰者类了，半装饰者在实际应用的比较少33

    • 不允许线程0重排序 使用 volatile 来修饰instance 来实现线程安全的初始化，重排序就会被禁止，在多线程的时候cpu也有共享内存，加了volatile关键字后，所有的线程都可以看到共享内存的最新状态，保证了内存的可见性，在用volatile修饰的变量在进行写操作的时候，会多出一些汇编代码，起到两个作用，第一，会将当前处理器缓存行的数据写回到内存中，这个写回内存的操作，会将其他cpu中缓存了该内存地址的数据无效，因为他们的数据无效了，所以他们又从共享内存中同步数据，这样就保证了内存的可见性，这里主要是使用缓存一致性协议，当处理器发现，我这个内存已经无效了，会重新将数据从共享内存中读到处理器的缓存中
    • 允许线程0重排序，不允许线程1看到线程0的重排序

    public class LazyDoubleCheckSingleton {

    private static volatile LazyDoubleCheckSingleton lazyDoubleCheckSingleton = null;
private LazyDoubleCheckSingleton(){
}

public static LazyDoubleCheckSingleton getInstance(){
      /*
        * 我们进行了两次 if (singleton == null) 检查，这就是“双重检查锁”这个名字的由来。
        * 这种写法是可以保证线程安全的，假设有两个线程同时到达 synchronized 语句块，那么实例化代码只会由其中先抢到锁的线程执行一次，
        * 而后抢到锁的线程会在第二个 if 判断中发现 singleton 不为 null，所以跳过创建实例的语句。再后面的其他线程再来调用 getInstance 方法时，
        * 只需判断第一次的 if (singleton == null) ，然后会跳过整个 if 块，直接 return 实例化后的对象。
        这种写法的优点是不仅线程安全，而且延迟加载、效率也更高。
        * */
    
       /*
            虽然判断了是否为空，但是有一种情况是，虽然他不为空，但是他并没有初始化完成，
            如果发生这种情况，就会返回一个未初始化完成的单例类，导致之后这个类的调用出现问题，
            使用volatile关键字可以保证即使你现在返回的是未初始化完成的单例类，
            但是当其他线程初始化完成之后，你之前得到的未初始化的类也会更新为初始化完成的单例类
            */
    if (lazyDoubleCheckSingleton == null){
        synchronized (LazyDoubleCheckSingleton.class){
            if (lazyDoubleCheckSingleton==null){
                lazyDoubleCheckSingleton = new LazyDoubleCheckSingleton();
                /*上面这一行看上去只是一行，实际上是三个步骤
                1. 分配内存给这个对象
                2. 初始化对象
                3. 设置lazyDoubleCheckSingleton 指向分配的内存地址
                2 和 3 的顺序有可能被颠倒
                intra-thread semantics 例如2，3的顺序被颠倒了也保证不会改变单线程内的执行结果
                解决方法:使用 volatile 关键字来修饰lazyDoubleCheckSingleton 防止重排序
                */
            }
        }
    }
    return lazyDoubleCheckSingleton;
}

  1. 定义:将一个复杂对象的构建与他的表示分离，使得同样的构建过程可以创建不同的表示
  2. 用户只需要指定需要构造的类型就可以得到他们，建造过程及细节不需要知道
  3. 就是如何一步一步建造一个包含多个组件的对象，相同的构建过程可以创建不同的产品，

适用场景

  1. 如果一个对象有非常复杂的内部结构（很多属性）
  2. 并且想把复杂对象的创建和适用分离

优点

  1. 封装性好，创建和适用分离
  2. 扩展性好，建造类之间独立，一定程度上解耦

缺点

  1. 产生多余的Builder对象
  2. 产品内部发生变化，建造者都需要修改，成本较大

和工厂模式比较

  1. 建造者模式更注重于方法的调用顺序，而工厂模式注重于创建产品
  2. 创建对象的力度不同，建造者模式可以创建复杂的产品，由各种复杂的部件组成，工厂模式创建出来的都是一个样子
  3. 关注点不同，工厂模式注重把这个对象创建出来就ok了，而建造者模式，不止要创建出这个产品，还要知道这个产品都是由哪些部件组成的
  4. 建造者模式再某些场景，比如，一定的顺序决定了产出的产品不一样，这里也要对顺序进行跳转，但是工厂并不关心顺序
  5. 当创建一个对象需要很多步骤时，适合适用建造者模式，而通过一个简单的方法就要创建出一个简单的对象时，适合适用工厂相关模式
